// What is a CSRF (Cross-Site Request Forgery) Attack?
A CSRF (Cross-Site Request Forgery) attack tricks a logged-in user into making unintended requests to a web application where they are authenticated.

The attacker exploits the user's session to perform malicious actions like changing passwords, transferring funds, or modifying data.

It usually works via hidden forms, malicious links, or embedded scripts.

1ï¸âƒ£ How Does a CSRF Attack Work?
Scenario: A user is logged into a banking website (bank.com), and an attacker sends them a malicious email containing:
html
Copy
Edit
<img src="https://bank.com/transfer?to=attacker&amount=1000" />
If the user clicks on the email while still logged into their bank, their browser automatically sends cookies with the request.

The bank processes the request as if it was made by the user.
âœ” Money is transferred without the userâ€™s consent!

2ï¸âƒ£ How to Prevent CSRF in Spring Boot?
Spring Security automatically protects against CSRF attacks for state-changing requests (POST, PUT, DELETE).

            .formLogin()
            .and()
            .logout();

    }

âœ” GET requests are allowed without a CSRF token since they do not modify data.

ğŸ”¹ Solution 2: Use CSRF Tokens in Forms

// Implementing Rate Limiting in Spring Boot
Rate limiting helps prevent DDoS attacks, API abuse, and excessive resource consumption by restricting the number of requests a client can make within a certain time frame.

1ï¸âƒ£ Using Spring Boot + Bucket4j (Token Bucket Algorithm)
Bucket4j is a popular in-memory rate-limiting library.

ğŸ”¹ Step 1: Add Dependency
xml
Copy
Edit

    <groupId>com.github.vladimir-bukhtoyarov</groupId>

</dependency>
ğŸ”¹ Step 2: Create a Rate Limiting Filter
java
Copy
Edit
import io.github.bucket4j.*;
import jakarta.servlet.http.HttpServletRequest;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.server.ResponseStatusException;

import java.io.IOException;
import java.time.Duration;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

@Component
public class RateLimitFilter extends OncePerRequestFilter {

    private final Map<String, Bucket> buckets = new ConcurrentHashMap<>();

    private Bucket createNewBucket() {
        return Bucket4j.builder()
                .addLimit(Bandwidth.classic(5, Refill.intervally(5, Duration.ofMinutes(1)))) // 5 requests per minute
                .build();
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request, jakarta.servlet.http.HttpServletResponse response, jakarta.servlet.FilterChain filterChain)
            throws IOException, jakarta.servlet.ServletException {
        
        String ip = request.getRemoteAddr(); // Identify the client by IP
        Bucket bucket = buckets.computeIfAbsent(ip, k -> createNewBucket());

        if (!bucket.tryConsume(1)) { // Consume 1 token per request
            response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            response.getWriter().write("Too many requests! Try again later.");
            return;
        }

        filterChain.doFilter(request, response);
    }
}
ğŸ”¹ Rate Limit: 5 requests per minute per IP.
ğŸ”¹ Refill Policy: 5 new requests allowed every minute.
ğŸ”¹ If limit is exceeded: Returns 429 Too Many Requests.

// Using Multiple Methods in Feign Client Interface
In a Feign client interface, you can define multiple methods for different HTTP operations (GET, POST, PUT, DELETE, etc.) for the same or different endpoints.

1ï¸âƒ£ Example Feign Client with Multiple Methods

Copy
Edit

import org.springframework.web.bind.annotation.*;

import java.util.List;

@FeignClient(name = "user-service", url = "http://localhost:8081")
public interface UserClient {

    // GET: Fetch user by ID
    @GetMapping("/users/{id}")
    UserResponse getUserById(@PathVariable("id") Long id);

    // GET: Fetch all users
    @GetMapping("/users")
    List<UserResponse> getAllUsers();

    // POST: Create a new user
    @PostMapping("/users")
    UserResponse createUser(@RequestBody UserRequest userRequest);

    // PUT: Update user details
    @PutMapping("/users/{id}")
    UserResponse updateUser(@PathVariable("id") Long id, @RequestBody UserRequest userRequest);

    // DELETE: Delete user by ID
    @DeleteMapping("/users/{id}")
    void deleteUser(@PathVariable("id") Long id);
}

// DTO for Request
class UserRequest {
    private String name;
    private String email;
    // Constructors, Getters & Setters
}

// DTO for Response
class UserResponse {
    private Long id;
    private String name;
    private String email;
    // Getters & Setters
}

 // When to Use REST API vs. Event-Driven Architecture
Both REST APIs and Event-Driven Architecture (EDA) are used for communication in microservices. Choosing between them depends on latency, scalability, coupling, and real-time processing needs.

ğŸ“Œ When to Use REST API (Synchronous Communication)
âœ” Simple Request-Response Use Cases â€“ When a client expects an immediate response.
âœ” CRUD Operations â€“ Best for managing users, orders, or products in a database.
âœ” Tightly Coupled Services â€“ When services directly depend on each other.
âœ” API Gateway & Load Balancing â€“ When central control over traffic is needed.
âœ” Small-scale systems â€“ Ideal for applications that donâ€™t require event streaming.

ğŸ”¹ Example: REST API Use Case

Fetching order details (GET /orders/{id})

Creating an order (POST /orders)

âŒ Limitations of REST
Blocking calls â€“ Can lead to high latency under heavy load.

Tightly coupled services â€“ A failure in one service may impact others.

Scalability issues â€“ Not ideal for real-time processing and event handling.

ğŸ“Œ When to Use Event-Driven Architecture (Asynchronous Communication)
âœ” Decoupled Microservices â€“ Services donâ€™t directly call each other, reducing dependencies.
âœ” Real-time Processing â€“ Best for handling streams of data (e.g., IoT, logs, notifications).
âœ” High Scalability â€“ Events are processed asynchronously, allowing parallel execution.
âœ” Fault Tolerance â€“ If a service fails, messages are stored in the queue and retried.
âœ” Bulk Data Processing â€“ Works well for batch jobs and analytics.

ğŸ”¹ Example: Event-Driven Use Case

User places an order â†’ Event order.created is published

Inventory Service listens and updates stock

Real-Time Analytics

Log events from different microservices are sent to Kafka for analysis.

Asynchronous Notifications

User signup â†’ Email & SMS notifications sent via event.

âŒ Limitations of Event-Driven
More Complex â€“ Requires message brokers like Kafka, RabbitMQ, SQS.

Eventual Consistency â€“ Services donâ€™t have instant data updates.

Debugging is Harder â€“ Since requests arenâ€™t direct, tracing failures is more difficult.

ğŸ“Œ REST vs Event-Driven: Quick Comparison
Feature	REST API	Event-Driven
Communication	Synchronous (Request-Response)	Asynchronous (Event-Based)
Coupling	Tightly Coupled	Loosely Coupled
Scalability	Limited	High
Latency	Higher	Lower
Real-Time Processing	âŒ No	âœ… Yes
Failure Handling	âŒ Harder (Retries needed)	âœ… Built-in Retry (Queues)
ğŸš€ Conclusion: When to Choose What?
